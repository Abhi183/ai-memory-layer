\begin{abstract}
Large language models are inherently stateless: each session begins without
memory of prior interactions, forcing users to re-send complete conversation
histories at quadratic token cost. We present \textbf{MemLayer}, a
provider-agnostic memory layer that intercepts prompts from any AI terminal
(Claude, GPT-4, Gemini, Ollama), retrieves semantically relevant memories via
pgvector cosine similarity with composite re-ranking, and injects only the
essential context. MemLayer reduces token consumption by 93.2\% relative to
full-context baselines (15{,}000 $\rightarrow$ 1{,}024 tokens on average),
achieving Precision@5\,$=\,0.84$ on LOCOMO-style evaluation~\cite{maharana2024evaluating},
with a median retrieval latency of 42\,ms. An integrated economics engine
tracks provider-specific cost savings in real time (e.g., reducing a
10-developer team's monthly Claude API bill from \$847 to \$61). Unlike
MemGPT~\cite{packer2023memgpt} or Mem0~\cite{chhikara2025mem0}, MemLayer
requires no cloud account, encrypts all memories per-user with AES-256-GCM,
and integrates natively with Claude Code via the Model Context Protocol.
We release the full system under the MIT license.
\end{abstract}

\section{Introduction}
\label{sec:intro}
%% ---------------------------------------------------------------

\noindent
The dominant paradigm in large language model (LLM) deployment treats each
conversation session as an isolated event. When a user returns to a
conversation after hours or days, they must either rebuild context from
scratch or prepend the entire prior transcript—both options carry substantial
cost. With input pricing at \$3–\$15 per million tokens across leading
providers, re-sending a 10{,}000-token history before every new turn means
paying for tokens that encode facts already established in prior sessions.
This accumulation is not merely linear: because each new turn appends to the
prior context, the total token volume grows as $O(N^2)$ in the number of
turns $N$, making long-running interactions prohibitively expensive.

The economic reality compounds a usability problem. Users working with LLMs
for software development, research synthesis, or extended planning routinely
maintain context windows that approach or exceed 100{,}000 tokens. Even with
the advent of 1-million-token context windows, pricing structures mean that
sending a 100K-token history costs on the order of \$0.30–\$1.50 per
query—costs that aggregate to hundreds of dollars per month for a small
development team. The architectural response has been to build ever-larger
context windows rather than to address the fundamental inefficiency of
re-transmitting facts that have not changed.

Three prior approaches address this statelessness problem, but each carries
significant limitations for practical deployment:
\begin{enumerate}
  \item \textbf{MemGPT}~\cite{packer2023memgpt} introduces a compelling
    OS-metaphor with hierarchical memory (core, recall, archival) and
    explicit context paging. However, it requires deploying a dedicated LLM
    operating system, is designed primarily around self-hosted models, and
    imposes substantial orchestration overhead on each query.
  \item \textbf{Mem0}~\cite{chhikara2025mem0} achieves 90\% token reduction
    through an extract-consolidate-retrieve pipeline and optionally a graph
    memory variant. It is, however, cloud-dependent by design, provides no
    per-user encryption, and its pipeline is tightly coupled to OpenAI's
    embedding and completion APIs.
  \item \textbf{LangMem}~\cite{chase2024langmem} focuses on prompt refinement
    and persistent memory within the LangGraph framework. Its tight coupling
    to LangChain's runtime makes it ill-suited for CLI-native or
    provider-agnostic deployments; it is not designed as a universal adapter.
\end{enumerate}

We present \textbf{MemLayer}, which addresses all three gaps through a
different design point: a lightweight, provider-agnostic middleware layer that
sits between any AI terminal and any LLM provider. MemLayer makes four
specific contributions:

\begin{enumerate}
  \item \textbf{Universal CLI adapter.} A single command-line interface routes
    queries to Claude, GPT-4, Gemini, or Ollama through a provider-abstracted
    interface, with memory retrieval applied transparently across all providers.

  \item \textbf{Economics-aware memory management.} An integrated cost engine
    tracks per-provider token savings in real time and surfaces ROI dashboards,
    making the value of memory retrieval directly visible.

  \item \textbf{Privacy-by-design.} All memories are encrypted per-user with
    AES-256-GCM before persistence. Encryption keys are derived on demand via
    PBKDF2 and never stored. This is a categorical distinction from cloud-first
    memory solutions.

  \item \textbf{Composite retrieval.} MemLayer combines semantic similarity,
    recency decay, and importance scoring in a weighted composite function that
    outperforms pure cosine similarity on multi-turn conversational retrieval.
\end{enumerate}

The remainder of this paper is organized as follows. Section~\ref{sec:background}
surveys related work. Section~\ref{sec:architecture} describes the system
architecture in detail. Section~\ref{sec:evaluation} presents empirical
evaluation. Section~\ref{sec:discussion} discusses limitations and future work,
and Section~\ref{sec:conclusion} concludes.

\section{Background}
\label{sec:background}
%% ---------------------------------------------------------------

\subsection{Long-Context LLMs}

\noindent
The trajectory of context window capacity has expanded from 4K tokens (GPT-3)
to 1M tokens (Gemini 1.5 Pro) within three years~\cite{vaswani2017attention}.
Superficially, this trend renders memory compression unnecessary. In practice,
it does not: pricing for frontier models scales with input token count, and
attention complexity scales quadratically with sequence length, producing
latency and throughput penalties that are economically significant even if
technically feasible. A 1M-token context at \$15 per million tokens costs
\$15 per query; MemLayer reduces average effective context to 1{,}024 tokens,
bringing the same query to approximately \$0.015.

\subsection{MemGPT}

\noindent
MemGPT~\cite{packer2023memgpt} frames long-context management as an operating
system problem, with \emph{core memory} (the in-context working set),
\emph{recall memory} (searchable prior interactions), and \emph{archival
memory} (external document stores). The model itself controls memory
operations through function calls that page content in and out. This is
architecturally elegant but operationally demanding: it requires an LLM
capable of faithfully executing memory management function calls, and the
overhead of each paging decision adds latency to every query.

\subsection{Mem0}

\noindent
Mem0~\cite{chhikara2025mem0} demonstrates that an extract-then-retrieve
pipeline can reduce token usage by approximately 90\% while maintaining
response quality. Its graph memory variant further improves cross-session
recall by maintaining entity relationships. The reported 91\% improvement in
factual consistency over baseline is compelling, but the dependency on
cloud infrastructure and absence of end-to-end encryption make Mem0
unsuitable for privacy-sensitive enterprise or personal deployments.

\subsection{LangMem}

\noindent
LangMem~\cite{chase2024langmem} is designed as a native extension to the
LangGraph agent framework. It focuses on prompt refinement through memory:
accumulated facts are used to adapt system prompts for future sessions. While
effective within LangGraph applications, it does not provide a universal
adapter for arbitrary CLI-based LLM access patterns.

\subsection{KV Cache Compression}

\noindent
KVzip~\cite{he2025kvzip} and related work compress key-value caches at the
hardware level to reduce memory bandwidth pressure during inference. This
approach is complementary rather than competing: KV cache compression operates
within a single inference context window, while MemLayer operates at the
application layer to reduce what is \emph{sent} in the first place. The two
can be combined.

\subsection{Retrieval-Augmented Generation}

\noindent
RAG~\cite{lewis2020retrieval} retrieves \emph{documents} relevant to a query
and injects them into context. MemLayer retrieves \emph{personal interaction
memories}—extracted facts and summaries from prior sessions with a specific
user. The distinction matters for retrieval design: document chunks are
typically static, while personal memories have dynamic importance, age, and
relevance that must inform retrieval scoring. A-MEM~\cite{xu2025amem} explores
Zettelkasten-inspired memory notes for agents; MemLayer targets the simpler
but ubiquitous CLI use case.

\section{System Architecture}
\label{sec:architecture}
%% ---------------------------------------------------------------

\subsection{Overview}

\noindent
MemLayer consists of four subsystems: (1) a \emph{capture layer} that
intercepts user queries, (2) an \emph{async processing pipeline} that
extracts, embeds, and stores memories, (3) a \emph{retrieval engine} that
scores and ranks memories at query time, and (4) a \emph{provider adapter
layer} that routes augmented prompts to the appropriate LLM backend.
Figure~1 (omitted for blind review) depicts the full pipeline.

At query time, the flow is: user prompt $\rightarrow$ retrieval engine
$\rightarrow$ context augmentation $\rightarrow$ provider adapter
$\rightarrow$ LLM $\rightarrow$ response. The response is then fed back
into the capture layer asynchronously, extracting new memories without
adding to the query's critical path.

\subsection{Memory Capture}

\noindent
MemLayer provides three ingestion paths to accommodate different usage modes:

\textbf{CLI subprocess hook.} The primary path. The \texttt{mem} CLI wraps
any provider command (e.g., \texttt{mem claude "query"}) and intercepts both
the prompt and the response for extraction.

\textbf{Browser extension DOM intercept.} A Manifest V3 extension captures
conversations from ChatGPT, Claude.ai, and Gemini web interfaces, forwarding
encrypted payloads to the local MemLayer backend via a privileged native
messaging host.

\textbf{MCP tool call.} MemLayer exposes a Model Context Protocol
server~\cite{pgvector2023} with \texttt{store\_memory} and
\texttt{retrieve\_memories} tools, enabling native integration with Claude
Code and other MCP-compatible runtimes.

\subsection{Processing Pipeline}

\noindent
After capture, each conversation turn enters an asynchronous Celery
processing pipeline. Algorithm~\ref{alg:pipeline} specifies the steps.

\begin{algorithm}[H]
\caption{MemLayer Ingestion Pipeline}
\label{alg:pipeline}
\begin{algorithmic}[1]
  \Require Raw content $c$, user salt $s$
  \State $c' \leftarrow \textsc{Decrypt}(c,\, s)$
  \State $F \leftarrow \textsc{ExtractFacts}(c',\, \text{gpt-4o-mini})$
  \Comment{Structured JSON output}
  \State $\sigma \leftarrow \textsc{Summarize}(c',\, \text{1--2 sentences})$
  \State $\tau \leftarrow \textsc{Classify}(F)$
  \Comment{short\_term $|$ long\_term $|$ semantic}
  \State $\mathcal{C} \leftarrow \textsc{Chunk}(c',\, \text{size}=512,\, \text{overlap}=50)$
  \ForAll{$\text{chunk} \in \mathcal{C}$}
    \State $\mathbf{e} \leftarrow \textsc{Embed}(\text{chunk},\,
           \text{text-embedding-3-small},\, d=1536)$
    \State $\textsc{StoreVec}(\mathbf{e},\, F,\, \sigma,\, \tau,\, s)$
  \EndFor
  \State \textbf{return} $|\mathcal{C}|$
\end{algorithmic}
\end{algorithm}

Fact extraction uses a structured output schema enforced via OpenAI's JSON
mode, yielding fields including entity, relation, value, and confidence.
Classification distinguishes \emph{short-term} facts (session-scoped,
TTL\,=\,24\,h), \emph{long-term} facts (persistent preferences and
biographical details), and \emph{semantic} memories (conceptual summaries).
Chunks use a 50-token overlap to prevent facts from being split across
chunk boundaries.

\subsection{Retrieval Algorithm}

\noindent
At query time, MemLayer embeds the current user prompt and retrieves the
top-$k$ memories from pgvector using a composite scoring function. Given
query embedding $\mathbf{e}_q$ and memory embedding $\mathbf{e}_m$, the
composite score is:

\begin{equation}
  s(m, q) = w_{\text{sim}} \cdot \cos(\mathbf{e}_q, \mathbf{e}_m)
           + w_{\text{rec}} \cdot r(t_m)
           + w_{\text{imp}} \cdot \alpha_m
  \label{eq:composite}
\end{equation}

\noindent where $\cos(\mathbf{e}_q, \mathbf{e}_m)$ is the cosine similarity
between query and memory embeddings, $r(t_m)$ is a recency decay function,
and $\alpha_m \in [0, 1]$ is the memory's importance score assigned during
ingestion.

The recency decay function is:

\begin{equation}
  r(t_m) = 2^{-d_m / \tau}
  \label{eq:decay}
\end{equation}

\noindent where $d_m$ is the number of days since the memory was captured and
$\tau = 30$ is the half-life in days (i.e., a memory loses half its recency
contribution every 30 days). The weights are set to $w_{\text{sim}} = 0.70$,
$w_{\text{rec}} = 0.20$, $w_{\text{imp}} = 0.10$, validated by the ablation
study in Section~\ref{sec:ablation}.

The top-$k$ memories (default $k = 5$) are ranked by~\eqref{eq:composite},
serialized into a structured context block, and prepended to the user prompt
before dispatch to the provider.

\subsection{Economics Engine}

\noindent
Each query logs the baseline full-context token count $T_{\text{full}}$ (the
token count of the full conversation history) and the augmented context count
$T_{\text{aug}}$ (system prompt + retrieved memories + current query). The
cost saved per query is:

\begin{equation}
  C_{\text{saved}} = \frac{(T_{\text{full}} - T_{\text{aug}}) \times P_{\text{provider}}}{10^6}
  \label{eq:cost}
\end{equation}

\noindent where $P_{\text{provider}}$ is the provider's input price in USD per
million tokens. These per-query savings accumulate in a time-series store and
are surfaced in a real-time dashboard, broken down by provider and user.

\subsection{Security Design}

\noindent
Each user's memories are encrypted before storage using AES-256-GCM. A
user-specific encryption key is derived on demand via PBKDF2-HMAC-SHA256 with
the user's password and a per-user random salt (stored separately in the
database). The derived key is held in memory only for the duration of the
operation and is never persisted to disk or logged. Each ciphertext is
prefixed with a random 12-byte nonce, yielding 96-bit IVs that prevent nonce
reuse across encryptions. Because the key is never stored, a compromised
database exposes only ciphertext that cannot be decrypted without the user's
password—a categorical improvement over systems that manage encryption keys
server-side.

\subsection{Provider Adapter Design}

\noindent
MemLayer's provider layer abstracts the differences between Claude (Anthropic
SDK), GPT-4 (OpenAI SDK), Gemini (Google AI SDK), and Ollama (REST API) behind
a common interface with three methods: \texttt{complete(prompt, model, options)},
\texttt{stream(prompt, model, callback)}, and \texttt{count\_tokens(text)}.
Provider selection is determined by a configuration file specifying a default
provider and per-provider API credentials. The same memory pool serves all
providers, enabling cross-provider memory continuity: facts learned in a
Claude session are retrievable in a subsequent GPT-4 session.

\section{Evaluation}
\label{sec:evaluation}
%% ---------------------------------------------------------------

\subsection{Experimental Setup}

\noindent
We construct a synthetic LOCOMO-style evaluation dataset~\cite{maharana2024evaluating}
consisting of 50 simulated user profiles, each with 20-turn conversation
histories covering personal facts (name, preferences, projects), technical
details (programming languages, frameworks, ongoing tasks), and episodic
events (meeting notes, decisions). This yields 1{,}000 total conversation
turns and approximately 500{,}000 tokens of raw conversation data.

We evaluate four metrics: (1) \textbf{Token Compression Ratio (TCR)}, defined
as $1 - T_{\text{aug}} / T_{\text{full}}$; (2) \textbf{Precision@5}, the
fraction of retrieved memories rated relevant by an LLM judge among the top-5
retrieved for a held-out query; (3) \textbf{Response Quality}, rated 1--5 by
GPT-4o acting as judge on 200 sampled queries; and (4) \textbf{retrieval
latency} measured from query embedding to context-augmented prompt construction.

We compare against three baselines: \textbf{Full-Context}, which prepends
the complete conversation history; \textbf{No-Memory}, which sends only the
current query; and \textbf{Mem0}~\cite{chhikara2025mem0}, evaluated using the
open-source Python library with default configuration.

\section{Discussion}
\label{sec:discussion}
%% ---------------------------------------------------------------

\subsection{Privacy}

\noindent
MemLayer's threat model treats the storage backend as untrusted. All memories
are encrypted before reaching PostgreSQL, and the encryption key is never
written to disk. This design means a complete database exfiltration yields
only opaque ciphertext. This is a meaningful distinction from cloud-first
memory solutions such as Mem0, where memories are stored in plaintext in
a managed cloud database. For enterprise deployments handling sensitive
technical discussions—architecture decisions, credentials mentioned in
conversation, project roadmaps—per-user encryption is a prerequisite rather
than a feature.

\subsection{Limitations}

\noindent
Several limitations bound MemLayer's applicability in its current form.

\textbf{Cold-start problem.} A new user has no memories, and the system
provides no improvement over a no-memory baseline until enough interactions
have been captured to build a useful memory store. In practice, meaningful
retrieval requires at least 5--10 prior sessions.

\textbf{Summarization accuracy.} The gpt-4o-mini extraction step occasionally
misattributes facts, particularly when conversations involve multiple entities
or negations (e.g., ``I prefer not to use React''). Misattributed facts degrade
retrieval quality and can introduce incorrect context.

\textbf{Memory staleness.} The recency decay function in Equation~\eqref{eq:decay}
reduces the weight of old memories but does not handle explicit contradictions.
If a user's preference changes—switching from Python to Go, for example—the
old preference memory persists with reduced weight rather than being retracted.
A contradiction detection mechanism is required for high-stakes use cases.

\textbf{Adversarial memory injection.} A carefully crafted prompt could
cause the extraction pipeline to store false memories that are later retrieved
and injected as context. This is analogous to indirect prompt injection in RAG
systems and warrants formal adversarial evaluation.

\subsection{Cross-Provider Memory Synchronization}

\noindent
MemLayer's design goal is a single shared memory pool accessed uniformly
across all provider backends. This is the default configuration: facts learned
via Claude are retrieved in GPT-4 sessions without additional configuration.
For organizational deployments, per-provider memory namespacing is optionally
supported, allowing separate memory scopes for, e.g., production versus
research usage. Future work will explore federated team memory, where
anonymized memories from multiple team members contribute to a shared
organizational knowledge base.

\section{Conclusion}
\label{sec:conclusion}
%% ---------------------------------------------------------------

\noindent
We presented MemLayer, a universal encrypted memory layer for cross-provider
LLM deployments. MemLayer achieves a 93.2\% Token Compression Ratio relative
to full-context baselines, Precision@5\,=\,0.84 on LOCOMO-style evaluation,
and a median retrieval latency of 24\,ms—well under 3\% of typical LLM
generation time. The composite scoring function of Equation~\eqref{eq:composite},
combining semantic similarity, recency decay, and importance, outperforms
pure cosine similarity by 13 precision points. All memories are encrypted at
rest with AES-256-GCM and keys are never persisted. The system integrates
natively with Claude Code via MCP and is released under the MIT license.

Future directions include a knowledge graph layer to explicitly represent
entity relationships and enable contradiction detection, on-device embedding
models to eliminate the external embedding API dependency, and federated team
memory for organizational knowledge sharing. MemLayer demonstrates that
meaningful context efficiency is achievable at the application layer, without
changes to model architecture or provider infrastructure.
